# Memory Strata — 四层分层记忆架构

## 为什么需要这个技能？

> "有记忆"和"没记忆"之间的差距，往往比不同 LLM 底座之间的差距还要大。
> —— arXiv:2603.07670

长程项目（持续数月到数年）中，Agent 普遍面临三大记忆问题：

1. **关键事实遗忘**：横屏/竖屏、模组选型等确定事实在会话间丢失
2. **记忆只增不减**：过时信息残留，MEMORY.md 膨胀
3. **知识不沉淀**：信息停留在每日日志中，未提升为可复用知识

本技能基于实践验证的四层架构解决这些问题。

## 核心架构：四层记忆分层

```
┌─────────────────────────────────────────┐
│  Layer 1: 系统注入层 (System Injection)  │  ← 每次会话自动加载
│  MEMORY.md — 核心要点 + 记忆锚点         │
├─────────────────────────────────────────┤
│  Layer 2: 项目核心层 (Project Core)      │  ← 极少变化
│  项目核心档案.md — 稳定事实 + ADR         │
├─────────────────────────────────────────┤
│  Layer 3: 开发状态层 (Dev Status)        │  ← 随开发更新
│  开发状态.md — 进展/参数/待做/已知问题     │
├─────────────────────────────────────────┤
│  Layer 4: 每日日志层 (Daily Journal)     │  ← 会话级
│  YYYY-MM-DD.md — 当天操作和调试过程       │
└─────────────────────────────────────────┘
```

### 各层职责

| 层级 | 内容 | 更新频率 | 位置 |
|------|------|----------|------|
| L1 系统注入 | 5-7条核心要点 + 记忆锚点 | 重要进展后 | `00-brain/MEMORY.md` |
| L2 项目核心 | 硬件规格、选型决策、ADR、架构 | 硬件/选型变更时 | `20-projects/<项目名>/项目核心档案.md` |
| L3 开发状态 | 进展清单、当前参数、已知问题 | 每次重要进展后 | `20-projects/<项目名>/开发状态.md` |
| L4 每日日志 | 当天操作、调试过程、临时发现 | 会话级 | `10-journal/YYYY-MM-DD.md` |

## 三大改进机制

### 机制1：自动蒸馏（Episodic → Semantic）

**蒸馏路径**：
```
L4 日志 ──[连续3天引用 或 里程碑完成]──→ L3 开发状态
L3 开发状态 ──[稳定≥1个月 且 确定事实]──→ L2 项目核心
L2 项目核心 ──[跨项目通用 或 用户偏好]──→ L1 系统注入
```

**提升条件**：
- L4→L3：里程碑完成、关键参数确定、操作方案稳定、连续>=2次引用
- L3→L2：稳定>=1个月 + 确定性事实 + 未来大概率需要
- L2→L1：对非本项目也有价值

**反模式**（避免）：
- 过度蒸馏：临时参数过早提升到 L2
- 蒸馏不足：关键事实永远停在 L4
- 跨层跳跃：日志直接跳到 L1

### 机制2：记忆强度衰减（last_used）

**规则**：
- MEMORY.md 中的每条决策/上下文附加 `[last_used: YYYY-MM-DD]` 标记
- 被检索命中或被使用时更新日期
- >14天未使用 → ⚡ 近期未使用，提示关注
- >30天未使用 → 考虑降级归档到 L3/L4

**用户偏好和经验教训类记忆**不衰减（长期有效）。

### 机制3：Obsidian 链接增强

**规则**：
- 各层文件之间通过 `[[]]` 双向链接
- L2↔L3 必须有链接（核心档案 ↔ 开发状态）
- 新增记忆时检查是否需与已有记忆建立链接
- 利用 Obsidian graph view 可视化知识结构

## 记忆锚点

对"易遗忘关键事实"的显式标记，在 L1 中强制加载。

**锚点格式**（在 MEMORY.md 的经验教训部分）：
```
- **[事实摘要]**：[详细说明]——[后果提示]
```

**示例**：
```
- **横屏是产品默认**：开发板默认竖屏，但产品横屏使用——这是项目的基本事实，不可遗忘
- **AP6256已选型**：SRRC已认证，是RK3568生态主流选择——不要重新研究这个问题
- **只用增量build.sh**：禁止--full全量编译——全量编译会失败浪费时间
```

## 使用方法

### 初始化（新项目首次使用）

1. 在 Obsidian vault 中创建 `20-projects/<项目名>/` 目录
2. 复制 `templates/` 下的模板文件到该目录
3. 填写项目核心档案中的稳定事实
4. 在 MEMORY.md 中添加项目摘要和锚点

### 日常维护

1. **会话开始**：`memory_search` 检索相关记忆，确认关键事实
2. **重要进展后**：更新 L3 开发状态
3. **选型/架构决策后**：更新 L2 核心档案的 ADR 表格
4. **会话结束前**：
   - 运行 `python scripts/memory_distill_check.py`
   - 根据提示执行蒸馏操作
   - 记录实践日志（如有失效事件或蒸馏操作）

### 蒸馏检查脚本

```bash
python scripts/memory_distill_check.py
```

输出5项检查：
1. 日志覆盖（近7天是否有日志）
2. 开发状态新鲜度（上次更新距今几天）
3. 核心档案新鲜度
4. 记忆强度衰减（last_used 标记检查）
5. 记忆锚点完整性
